Decision — Filtering at Launch
Option A — Launch with filtering
Pills bar appears above the resource grid from day one. Future-proofed for when the library grows. Slightly more build.
Option B — Launch without, add later
Simple grid only. When Stephanie has 8+ resources, we add filtering then. Cleanest for v1.
Recommendation: Option B for v1 unless Stephanie has a backlog of resources ready to load on launch day.

Three new fields on the event document: Collect Interest (toggle), Notification Email (where to send the heads-up), and Interested Emails (the captured list). ✓ Confirmed Decisions
  • ✅
    No On-Site Payment Processing — All resource cards link to external destinations (Amazon, Eventbrite, direct PDF download, etc.). The Resources page is a curated link directory, not a shop. No checkout, no file uploads, no Lemon Squeezy, no Stripe.
  • ✅
    Resources on Both Sides — Stephanie tags each resource for Individual, Business, or both. Single CMS document — no duplicate entries.
  • ✅
    Conditional Nav — The Resources nav link only appears on a side when at least one resource exists for that side. Empty sides stay clean — no broken-looking nav.
  • ✅
    Event Interest Capture is Opt-In Per Event — Stephanie chooses which events have the signup form via a toggle on the event document. Off by default.
  • ✅
    Notification Email on Signup — Stephanie gets an email whenever someone joins an interest list. No need to check the CMS just to see if anyone signed up.
  • ✅
    Returning Client Engagement — Separate scope from the launch contract and content support. Treated as a new mini-project with its own deposit, timeline, and deliverable.
  • ✅
    Future Flexibility — On-site payment processing (Lemon Squeezy, Stripe, or otherwise) can be added as a separate engagement later. Nothing about this build prevents that path.
